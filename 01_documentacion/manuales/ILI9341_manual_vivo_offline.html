2) Hardware mínimo recomendado

El ILI9341 normalmente trabaja a lógica 3.3V. Si tu MCU es 5V, usa adaptación de nivel.
Conecta siempre: VCC, GND, SCK, MOSI, CS, DC, RST. MISO es opcional en muchos módulos.
Sin desacoplo cercano (100nF + 1uF), tendrás fallos intermitentes y ruido visual.
// Pines típicos (ajusta a tu placa)
// SPI: SCK, MOSI
// Control: CS, DC, RST
// Backlight: BL (PWM opcional)

3) SPI y comandos base

El flujo básico es: seleccionar CS, enviar comando con DC=0, enviar datos con DC=1, deseleccionar CS.

static inline void tft_cmd(uint8_t c){
  DC_LOW(); CS_LOW(); spi_tx(&c, 1); CS_HIGH();
}

static inline void tft_data(const uint8_t *d, size_t n){
  DC_HIGH(); CS_LOW(); spi_tx(d, n); CS_HIGH();
}

// Comandos típicos
// 0x01 SWRESET, 0x11 SLPOUT, 0x29 DISPON
// 0x2A CASET, 0x2B PASET, 0x2C RAMWR

4) Inicialización mínima (nivel principiante)

Secuencia robusta mínima para encender pantalla y llenar color.

void ili9341_init(void){
  RST_LOW(); delay_ms(20);
  RST_HIGH(); delay_ms(120);

  tft_cmd(0x01);              // SWRESET
  delay_ms(120);
  tft_cmd(0x11);              // SLPOUT
  delay_ms(120);

  uint8_t pixfmt = 0x55;      // 16-bit/pixel (RGB565)
  tft_cmd(0x3A); tft_data(&pixfmt, 1);

  tft_cmd(0x29);              // DISPON
  delay_ms(20);
}

5) Dibujar píxel, ventana y rectángulo

void set_window(uint16_t x0, uint16_t y0, uint16_t x1, uint16_t y1){
  uint8_t d[4];
  tft_cmd(0x2A); // CASET
  d[0]=x0>>8; d[1]=x0; d[2]=x1>>8; d[3]=x1; tft_data(d,4);
  tft_cmd(0x2B); // PASET
  d[0]=y0>>8; d[1]=y0; d[2]=y1>>8; d[3]=y1; tft_data(d,4);
  tft_cmd(0x2C); // RAMWR
}

void fill_rect(uint16_t x, uint16_t y, uint16_t w, uint16_t h, uint16_t c){
  set_window(x,y,x+w-1,y+h-1);
  DC_HIGH(); CS_LOW();
  for(uint32_t i=0;i<(uint32_t)w*h;i++){
    uint8_t px[2] = { c >> 8, c & 0xFF };
    spi_tx(px,2);
  }
  CS_HIGH();
}

6) Texto e imágenes (nivel intermedio)

Texto básico

Renderiza glifos 1bpp y pinta foreground/background en RGB565.

Recomendado para UI simple

Bitmaps

Convierte imágenes a RGB565 y envía por bloques para no agotar RAM.

Usa chunks de 1-4 KB
// Conversión rápida 24-bit a RGB565
static inline uint16_t rgb565(uint8_t r, uint8_t g, uint8_t b){
  return ((r & 0xF8) << 8) | ((g & 0xFC) << 3) | (b >> 3);
}

7) Rendimiento con DMA (nivel avanzado)

Para FPS altos, evita mandar píxel por píxel desde CPU. Usa buffers y DMA SPI.

// Idea general (pseudocódigo)
// 1) Reservar buffer de línea o bloque
// 2) Rellenar buffer en RAM
// 3) Lanzar spi_tx_dma(buffer, n)
// 4) Esperar callback half/full complete

void flush_block_dma(uint16_t x, uint16_t y, uint16_t w, uint16_t h, const uint16_t *buf){
  set_window(x,y,x+w-1,y+h-1);
  DC_HIGH(); CS_LOW();
  spi_tx_dma((const uint8_t*)buf, (size_t)(w*h*2));
  // CS_HIGH en callback de fin
}
En DMA, el cuello de botella suele estar en SPI clock, tamaño de bloque y sincronización de callbacks.

8) Integración con LVGL (nivel experto)

// lvgl flush callback (esquema)
void my_flush_cb(lv_display_t *disp, const lv_area_t *area, uint8_t *px_map){
  uint16_t w = (uint16_t)(area->x2 - area->x1 + 1);
  uint16_t h = (uint16_t)(area->y2 - area->y1 + 1);
  flush_block_dma((uint16_t)area->x1, (uint16_t)area->y1, w, h, (uint16_t*)px_map);
  // Llamar lv_display_flush_ready(disp) al terminar DMA
}

Buenas prácticas: doble buffer parcial, tick estable, y nunca bloquear flush durante demasiado tiempo.
