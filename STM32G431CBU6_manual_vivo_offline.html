00C) Electricidad aplicada base (lo que conviene dominar)

Antes de configurar perifericos, necesitas controlar estas reglas electricas. Muchos fallos "de firmware" son en realidad problemas de nivel de tension, corriente o disipacion.

Magnitudes basicas y formulas utiles

V = I * R
I = V / R
P = V * I = I^2 * R = V^2 / R
Vout (divisor) = Vin * (R2 / (R1 + R2))
Concepto Que debes recordar Riesgo si lo ignoras
Niveles logicos STM32G431 trabaja tipicamente a 3.3 V en GPIO. Aplicar 5 V directos en pin no tolerante puede dañar el MCU.
Corriente por pin Respetar limites por pin y total de puerto/dispositivo (datasheet). Sobrecalentamiento, reset o daño permanente.
Resistencia en serie Limita corriente y ayuda a proteger entradas/salidas. Picos de corriente excesivos y señales degradadas.
Pull-up / Pull-down Evita pines flotantes y lecturas aleatorias. Triggers falsos e inestabilidad logica.
Potencia de resistencias Comprueba P disipada y margen de seguridad (x2 recomendado). Resistencia caliente o quemada.
Practica segura: cuando tengas duda de niveles o corriente, vuelve al datasheet y calcula. Nunca asumas compatibilidad de voltajes "porque funciona a veces".

01) Vision general del MCU

Cortex-M4F Hasta 170 MHz DSP + FPU FMAC/CORDIC ADC rapido Timers avanzados

STM32G431 esta orientado a control en tiempo real y aplicaciones de signal mixed. Es muy buen candidato para instrumentacion, control de potencia, motor y medicion rapida.

Puntos fuertes

  • Muy buena combinacion de perifericos analogicos y timers.
  • Rendimiento suficiente para DSP basico en tiempo real.
  • Ecosistema robusto (CubeIDE, HAL/LL, ST-LINK, comunidad).

Riesgos tipicos

  • No entender clock tree y romper tiempos de muestreo.
  • Confiar en defaults sin revisar registros y errata.
  • Ignorar layout analogico y culpar al firmware por ruido.

02) Arranque, startup y mapa de memoria

Flujo de arranque

Reset -> Vector Table -> Reset_Handler (startup_stm32g4xx.s)
      -> SystemInit() (clock/base)
      -> __libc_init_array() (C runtime)
      -> main()

Bloques de memoria (conceptual)

Bloque Contenido Consejo
Flash Codigo, const, tablas, vector table Evita escrituras frecuentes sin estrategia de wear.
SRAM Variables globales, heap, stack, buffers DMA Dimensiona stack segun ISR y printf.
CCM SRAM (si aplica) Memoria rapida para rutinas/datos criticos Ubica buffers o codigo de baja latencia si procede.
Backup domain RTC y registros backup con VBAT Guarda contadores/flags de recuperacion.

Boot y actualizacion

Para bootloader de sistema, revisa AN de ST para interfaces soportadas por variante/pines de boot. Para producto serio, considera bootloader propio con verificacion CRC y rollback.

03) Clock tree y PLL

El reloj define todo: temporizacion de CPU, timers, ADC y buses.

Estructura base

HSI/HSE ----> PLL(M,N,R,P,Q) ----> SYSCLK ----> AHB(HCLK) ----> APB1/APB2(PCLK)
                                               \-> Perifericos (TIM, ADC, SPI...)

Formulas de PLL (conceptuales)

fVCO_in = fPLL_source / M
fVCO_out = fVCO_in * N
fPLL_R = fVCO_out / R
Los rangos validos de M/N/R y VCO dependen de RM0440 y la revision concreta. Usa siempre la herramienta de Clock Configuration de CubeMX y valida con RM.

Relacion APB y timers

Segun prescaler APB, el reloj efectivo de timers puede no ser igual al PCLK. Este punto causa muchos errores de frecuencia. Compruebalo siempre en Cube y RM.

04) GPIO y EXTI

Modos de GPIO

Modo Uso tipico Error frecuente
Input Lectura de sensores/binarios No configurar pull y dejar pin flotando.
Output PP Control directo de linea digital Conducir carga excesiva sin driver.
Output OD Buses compartidos (I2C) o lineas abiertas Olvidar pull-up.
AF SPI/I2C/UART/TIM Escoger AF incorrecta para ese pin.
Analog ADC, reducir consumo en pin no usado Dejar un pin de comunicacion en analog.

EXTI robusto

  • Configura flanco correcto (rising/falling/both).
  • Haz debounce (software/hardware).
  • En ISR: solo bandera y salida rapida.

05) NVIC e interrupciones

Prioridades bien elegidas = sistema estable. ISR largas = jitter y perdida de eventos.

Prioridad sugerida Tipo Motivo
Alta ADC DMA complete, safety critical No perder datos ni eventos criticos.
Media Timer control, captura auxiliar Control temporal estable.
Baja UI, UART logs, tareas no criticas Evitar bloquear cadena de adquisicion. 
// Patron recomendado de ISR

void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef *hadc)
{
    if (hadc->Instance == ADC1) {
        g_evt_adc_done = 1; // solo evento
    }
}

06) Timers: tipos, bits, registros y reloj

Tipos de timer (serie G4, depende de variante/paquete)

Categoria Ejemplos Bits Uso
Basicos TIM6, TIM7 16-bit Base temporal y trigger de ADC.
Generales TIM2, TIM3, TIM4... TIM2 suele 32-bit; otros 16-bit PWM, input capture, encoder.
Avanzados TIM1/TIM8 segun variante 16-bit Complementary PWM, dead-time, break.

Esquema de registros clave de timer

TIMxCLK -> [PSC] -> [CNT] <-> [CCR1..CCR4]
                     |
                     +--> compara contra ARR -> update event(UIF)

Registros clave:
- CR1   : CEN, DIR, ARPE...
- PSC   : prescaler
- ARR   : auto reload
- CNT   : contador actual
- DIER  : habilita IRQ/DMA
- SR    : flags (UIF, CCxIF...)
- EGR   : genera update por software (UG)
- CCMR* : modos output/input compare
- CCER  : habilita/cfg canales
- BDTR  : dead-time/break (timers avanzados)

Frecuencia de update

f_cnt = f_TIMxCLK / (PSC + 1)
f_update = f_cnt / (ARR + 1)
T_update = 1 / f_update 
// Ejemplo: 170 MHz -> 5 kHz
// PSC=169 => f_cnt=1 MHz
// ARR=199 => f_update=1 MHz/200=5 kHz

07) PWM, Input Capture y Encoder mode

PWM

Ideal para control de potencia y actuadores. La resolucion depende de ARR: a mayor ARR, mejor granularidad, pero limitada por frecuencia objetivo.

duty(%) = (CCR / (ARR + 1)) * 100

Input Capture

Mide periodo/frecuencia capturando CNT en flancos de entrada.

periodo_ticks = capture_n - capture_{n-1}
f_signal = f_cnt / periodo_ticks

Encoder mode

Usa dos canales en cuadratura para posicion y velocidad en encoders incrementales.

Para control de motor o potencia, revisa timers avanzados con dead-time y break. Son diferenciales respecto a MCUs mas simples.

08) ADC, analogico y DMA

Arquitectura recomendada de adquisicion

Timer TRGO -> ADC conversion -> DMA -> Buffer RAM -> Procesado en loop/tarea

Puntos tecnicos criticos

  • Sampling time: depende de impedancia de fuente; muy corto puede distorsionar.
  • Calibracion ADC: ejecutala al iniciar para reducir error.
  • Oversampling: mejora ENOB efectiva a costa de tasa de salida.
  • Analog watchdog: detecta umbrales sin CPU pesada.

Cadena de conversion

V = (counts / FS_counts) * Vref
F = gain * (V - offset)
Si la aplicacion requiere precision metrologica elevada, un ADC externo dedicado + referencia estable suele superar al ADC interno en ruido y linealidad global del sistema.

09) DAC, comparadores, OPAMP y VREFBUF

La serie G4 integra bloques analogicos muy utiles para prototipos y control rapido.

Bloque Uso Ventaja
DAC Generar referencia/senal de prueba Validacion rapida sin equipo externo en ciertos escenarios.
Comparadores Deteccion de umbral hardware Respuesta rapida y baja carga de CPU.
OPAMP Buffer o PGA interno Ahorra componentes externos para prototipos.
VREFBUF Referencia interna configurable Mejora consistencia de conversion si se usa bien.
Para diseno final de precision: valida offset/ruido/deriva termica de cada bloque analogico con pruebas reales y caracterizacion de temperatura.

10) Comunicaciones: USART, SPI, I2C y mas

USART/UART

  • Debug principal del firmware.
  • Con DMA RX circular puedes crear consola robusta.
  • Valida error de baud con clock real.

SPI

  • Ideal para ADC externo y pantallas.
  • Revisa CPOL/CPHA y timing CS.
  • DMA recomendado para transferencias largas.

I2C

  • Sensores lentos/medios y perifericos de control.
  • Pull-up correcto es obligatorio.
  • Implementa recuperacion de bus colgado.

Otros (segun variante)

  • FDCAN para redes robustas.
  • USB FS para conectividad con PC.
  • LPUART para bajo consumo.

11) RTC, backup y modos de bajo consumo

Modo Consumo relativo Latencia wakeup Uso tipico
Sleep Bajo Muy baja Espera corta conservando estado completo.
Stop Muy bajo Media Duty-cycle con RTC/EXTI.
Standby/Shutdown Minimo Mayor (reinicio parcial/total) Autonomia maxima.

Usa VBAT para preservar RTC y backup registers. Muy util para contador de eventos, marca temporal y recuperacion tras corte.

12) Watchdogs y resiliencia del firmware

Tipos

  • IWDG: independiente, robusto frente a fallo de clock principal.
  • WWDG: ventana temporal; detecta tanto retraso como refresco demasiado temprano.

Patron recomendado

  • Alimentar watchdog solo cuando el ciclo critico complete correctamente.
  • No hacer refresh en ISR indiscriminadamente.
  • Guardar causa de reset y estado previo en backup para diagnostico.

13) Registros: metodo practico para no perderte

Secuencia mental universal

  1. Habilita clock del periferico en RCC.
  2. Configura modo/parametros en registros de configuracion.
  3. Genera update/evento inicial si aplica.
  4. Limpia flags pendientes.
  5. Habilita interrupcion/DMA.
  6. Arranca periferico.
// Ejemplo minimo conceptual timer
RCC->APB1ENR1 |= RCC_APB1ENR1_TIM6EN;
TIM6->PSC = 169;
TIM6->ARR = 199;
TIM6->EGR = TIM_EGR_UG;
TIM6->SR = 0;                 // limpia flags
TIM6->DIER |= TIM_DIER_UIE;
TIM6->CR1 |= TIM_CR1_CEN;
Evita escribir registros sin revisar descripcion de bit, side effects y orden de actualizacion en RM0440.

14) HAL vs LL vs registros directos

Enfoque Pros Contras Cuando usar

LL Mas cercano a HW, mejor rendimiento Mas detalle y complejidad Perifericos criticos
Registros Control total y latencia minima Mantenimiento dificil y mayor riesgo de error Secciones de timing extremo o drivers propios

16) Seguridad basica y proteccion de firmware

  • RDP: readout protection para frenar extraccion de firmware.
  • WRP: write protection por paginas de Flash.
  • PCROP: protege lectura de codigo propietario.
Define la politica de seguridad temprano. Cambiar niveles de proteccion en fases tardias puede complicar depuracion y produccion.

17) Hardware y PCB para STM32

Reglas cortas de diseno

  • Desacoplo cercano a cada pin VDD/VDDA.
  • Plano de masa coherente y rutas de retorno claras.
  • Separar zonas analogicas de digitales ruidosas.
  • Mantener SWD accesible (header o pads).
  • Protecciones ESD en conectores externos.
Problema Sintoma Mitigacion
Ruido en ADC Lecturas inestables Mejor GND, filtro RC, referencia limpia, layout.
SPI inestable Frames corruptos Reducir slew/ruta, ajustar speed y terminacion segun trazas.
Reset aleatorio Reinicio bajo carga Revisar BOR/PDR, desacoplo y transitorios de alimentacion.

19) Recetas rapidas de firmware

R1: Blink no bloqueante

Usa tick por timer, evita HAL_Delay en logica principal.

R2: UART debug estable

TX por DMA o anillo; evita printf en ISR.

R3: Timer 5 kHz

Base para trigger ADC y procesos periodicos.

R4: ADC + DMA

Captura N muestras con trigger por timer.

R5: Input capture

Mide frecuencia de sensor digital rapido.

R6: PWM variable

Control de actuador con duty configurable.

R7: I2C sensor

Lectura periodica con validacion de timeout.

R8: SPI display

Dibujo por bloques para no bloquear tiempo critico.

R9: Watchdog seguro

Refresh solo tras ciclo valido completo.

// Esqueleto base: maquina de estados
switch (state) {
  case ST_INIT:    init_all(); state = ST_READY; break;
  case ST_READY:   if (trigger()) state = ST_CAPTURE; break;
  case ST_CAPTURE: start_capture_dma(); state = ST_WAIT; break;
  case ST_WAIT:    if (capture_done) state = ST_PROCESS; break;
  case ST_PROCESS: process(); state = ST_DISPLAY; break;
  case ST_DISPLAY: draw(); if (reset_req) state = ST_READY; break;
  default:         state = ST_ERROR; break;
}

Normal Captura puntual de N muestras Simple de razonar Se detiene al finalizar

Half/Full callback Procesado por bloques Menor latencia de procesado Complejidad de sincronizacion

  evt_half = 1;
}

  evt_full = 1;
}

23) Linker script y organizacion de firmware

El linker script define donde vive cada seccion. Entender esto evita fallos extranos por stack/heap y permite ubicar buffers criticos.

Secciones comunes

Seccion Contenido Carga
.text Codigo ejecutable Flash
.rodata Constantes Flash
.data Globales inicializadas Flash -> SRAM en arranque
.bss Globales a cero SRAM
Si usas DMA, evita ubicar buffers en memorias no accesibles por DMA segun arquitectura. Revisa mapeo de buses y restricciones en RM.

24) Bootloader y actualizacion de firmware

Opciones

  • Bootloader de sistema ST (segun interfaz y pines disponibles).
  • Bootloader propio en Flash para control total de protocolo y seguridad.

Patron robusto de actualizacion

  1. Descargar imagen en zona temporal.
  2. Verificar integridad (CRC/hash y version).
  3. Marcar estado "pending" en backup register.
  4. Conmutar a nueva imagen.
  5. Si falla self-test, rollback automatico.
Nunca sobrescribas la unica imagen valida sin mecanismo de recuperacion. Es la causa principal de equipos "brick" en campo.

26) Flujo de herramientas y depuracion

Pipeline recomendado

  1. CubeMX/CubeIDE para pinout y clock base.
  2. Compilacion con warnings altos y sin warnings nuevos.
  3. Flash con ST-LINK y validacion de startup.
  4. Pruebas de tiempo con osciloscopio/analizador logico.
  5. Test de estres (horas) con watchdog habilitado.
Mejora de productividad: usa un "test bench" minimo por periferico (timer, ADC, UART, SPI) antes de integrar todo en una sola app.

Atajos utiles de depuracion

F5 continuar, F10 step over, F11 step into, watch expressions para registros clave (RCC, TIMx, ADCx, DMA).
